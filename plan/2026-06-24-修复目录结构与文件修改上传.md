# 修复定时任务与文件监控的目录结构 + 文件修改自动上传

**日期**：2026-06-24

## Context

测试发现两个问题：
1. 定时任务选择文件夹 `/Users/liguorui/Desktop/备份`，服务器路径应为 `/home/备份/file.txt`，实际为 `/home/file.txt`（丢失了文件夹名"备份"）
2. 文件监控同样的目录结构问题 + 文件修改后未自动上传

## 根因分析

### 目录结构问题

`scheduler.ts` 的 `buildRemotePath` 和 `file-watcher.ts` 的 `handleNewFile` 都用 `path.relative(sourcePath, filePath)` 计算相对路径，会丢掉源文件夹自身的名称。

示例：sourcePath = `/Users/liguorui/Desktop/备份`, file = `/Users/liguorui/Desktop/备份/test.txt`
- `path.relative(sourcePath, file)` = `test.txt`
- 期望结果：`备份/test.txt`（保留文件夹名）

修复：用 `path.dirname(sourcePath)` 作为 `path.relative` 的基准，这样源文件夹自身的名称会保留在相对路径中。

### 文件修改不上传

`file-watcher.ts` 的 `addWatcher` 已注册 `change` 事件，但 chokidar 的 `awaitWriteFinish` 配置可能导致检测问题。另外需要确认 `filePattern` 匹配逻辑不会误过滤。

## 改动计划

### 1. scheduler.ts — buildRemotePath 保留源文件夹名

**文件:** `src/main/services/scheduler.ts`

```ts
// 修复前
const relativePath = baseSourcePath
    ? path.relative(baseSourcePath, localPath)
    : path.basename(localPath)

// 修复后
const relativePath = baseSourcePath
    ? path.relative(path.dirname(baseSourcePath), localPath)
    : path.basename(localPath)
```

### 2. file-watcher.ts — handleNewFile 保留监控文件夹名

**文件:** `src/main/services/file-watcher.ts`

```ts
// 修复前
const relativePath = path.relative(config.watchPath, filePath)

// 修复后
const relativePath = path.relative(path.dirname(config.watchPath), filePath)
```

### 3. file-watcher.ts — 确保 change 事件正常工作

确认 `addWatcher` 中 `change` 事件已注册，检查 `awaitWriteFinish` 配置是否影响 `change` 事件触发。可能需要调整 `stabilityThreshold` 或在 `change` 事件中使用不同的 debounce 策略。

## 改动文件清单

| 文件 | 改动 |
|------|------|
| `src/main/services/scheduler.ts` | buildRemotePath 用 `path.dirname` 保留文件夹名 |
| `src/main/services/file-watcher.ts` | handleNewFile 用 `path.dirname` 保留文件夹名 |

## 验证

1. 定时任务：sourcePath=`/Users/liguorui/Desktop/备份`, remotePath=`/home` → 上传 test.txt 到 `/home/备份/test.txt`
2. 文件监控：watchPath=`/Users/liguorui/Desktop/备份`, remotePath=`/home` → 新文件上传到 `/home/备份/xxx`
3. 文件监控：修改已有文件 → 自动重新上传
