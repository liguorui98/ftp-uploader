# 修复下载取消功能

## Context

点击"取消下载"无反应。根本原因：在 `PassThrough.on('data')` 回调中 throw 不会中断底层 FTP/SFTP 下载 — `await downloadTo()` / `await get()` 不受影响。且 cancel handler 没有活跃客户端的引用，无法断开连接。

## 修复方案

用 `Map<string, { client, passThrough, localPath }>` 存储活跃下载的引用。cancel 时主动 destroy 流 + disconnect 客户端，导致 await 拒绝，再清理临时文件。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/main/ipc/index.ts` | 用 Map 替代 Set 存储活跃下载引用；cancel 时 destroy 流 + disconnect；下载结束清理引用和临时文件 |

仅 1 个文件。

---

## 具体改动

### `src/main/ipc/index.ts`

#### 1. 替换 cancelledDownloads 为 activeDownloads Map

```ts
// 替换:
const cancelledDownloads = new Set<string>()

// 为:
const activeDownloads = new Map<string, {
  client: TransferClient
  passThrough: PassThrough
  localPath: string
}>()
```

#### 2. 修改 browser:download — 注册/注销活跃下载

- 创建 client 后、调用 download 前，将 `{ client, passThrough, localPath }` 存入 activeDownloads
- download 完成/失败后，从 activeDownloads 中删除
- 在 onProgress 中检查：如果 downloadId 不在 activeDownloads 中（说明已被取消），throw 中断

#### 3. 修改 browser:cancel-download — 主动断开

```ts
ipcMain.handle('browser:cancel-download', async (_, downloadId: string) => {
  const active = activeDownloads.get(downloadId)
  if (active) {
    active.passThrough.destroy(new Error('下载已取消'))
    await active.client.disconnect().catch(() => {})
    try { fs.unlinkSync(active.localPath) } catch {}
    activeDownloads.delete(downloadId)
  }
  return { success: true }
})
```

#### 4. browser:download 的 catch 中增加临时文件清理

```ts
catch (error) {
  try { fs.unlinkSync(result.filePath) } catch {}
}
```

---

## 验证

1. 点击下载 → 进度弹框显示 → 点击"取消下载" → 下载立即中断，弹框关闭
2. 取消后本地无残留临时文件
3. 取消后可以正常再次下载同一文件
4. 下载完成后正常关闭弹框，无异常

# 第二轮修复

## Context

第一轮修复后测试仍无效。根本原因：IPC handler 创建的 `passThrough` 存储在 `activeDownloads` 中，但客户端 `download()` 方法内部创建了**自己的** `passThrough`。cancel handler destroy 的是 IPC 层的 passThrough，客户端内部的 passThrough 不受影响。

## 修复方案

在 `TransferClient.download()` 接口新增 `AbortSignal` 参数。cancel 时通过 `AbortController.abort()` → signal abort → 客户端内部 destroy passThrough → downloadTo/get reject。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/main/services/ftp-client.ts` | TransferClient.download 新增 signal 参数；FTPClient/SFTPClient download 监听 signal abort 时 destroy passThrough |
| `src/main/ipc/index.ts` | 用 AbortController 替代 passThrough；cancel 时调用 abortController.abort() |

仅 2 个文件。

---

## 步骤 1：ftp-client.ts — download 支持 AbortSignal

### TransferClient 接口

```ts
download(remotePath: string, localPath: string, onProgress?: ProgressCallback, signal?: AbortSignal): Promise<void>
```

### FTPClient.download（约 line 142）

在 `onProgress && size > 0` 分支中，创建 passThrough 后注册 signal 监听：

```ts
if (signal) {
  if (signal.aborted) {
    passThrough.destroy(new Error('下载已取消'))
  } else {
    signal.addEventListener('abort', () => {
      passThrough.destroy(new Error('下载已取消'))
    }, { once: true })
  }
}
```

### SFTPClient.download（约 line 381）

同样的模式。

---

## 步骤 2：ipc/index.ts — 用 AbortController 替代 passThrough

### activeDownloads Map 类型改为

```ts
const activeDownloads = new Map<string, {
  client: TransferClient
  abortController: AbortController
  localPath: string
}>()
```

### browser:download handler

```ts
const abortController = new AbortController()
activeDownloads.set(downloadId, { client, abortController, localPath: result.filePath })
await client.download(remotePath, result.filePath, onProgress, abortController.signal)
```

### browser:cancel-download handler

```ts
const active = activeDownloads.get(downloadId)
if (active) {
  active.abortController.abort()
  activeDownloads.delete(downloadId)
}
```

不需要手动 disconnect — signal abort 会 destroy passThrough，导致 downloadTo/get reject，finally 块会处理 disconnect。

# 第三轮修复

## Context

两轮修复后仍无效。根本原因：`passThrough.destroy()` 不会中断底层 FTP/SFTP 库的 socket 读取。`basic-ftp` 的 `downloadTo()` 和 `ssh2-sftp-client` 的 `get()` 继续从 socket 读数据，流被 destroy 只是丢弃数据，不会 reject promise。

**唯一能中断传输的方式是关闭底层连接**：`this.client.close()`（FTP）/ `this.client.end()`（SFTP）。

## 修复方案

将断开连接的回调传入 `download()` 方法，signal abort 时直接关闭底层连接。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/main/services/ftp-client.ts` | download 新增 `cancelClient?` 参数；signal abort 时调用 `cancelClient()` 关闭底层连接 |
| `src/main/ipc/index.ts` | 传入 `() => client.disconnect()` 作为 cancelClient |

---

## ftp-client.ts

### TransferClient 接口

```ts
download(remotePath: string, localPath: string, onProgress?: ProgressCallback, signal?: AbortSignal, cancelClient?: () => Promise<void>): Promise<void>
```

### FTPClient.download / SFTPClient.download

signal abort 时调用 `cancelClient()` 关闭底层连接：

```ts
if (signal) {
  const onAbort = () => { cancelClient?.().catch(() => {}) }
  if (signal.aborted) { onAbort() }
  else { signal.addEventListener('abort', onAbort, { once: true }) }
}
```

## ipc/index.ts

```ts
await client.download(remotePath, result.filePath, onProgress, abortController.signal,
  async () => { await client.disconnect().catch(() => {}) }
)
```

2. 取消后无残留临时文件
3. 取消后可再次下载

# 第四轮修复（根因修复）

## Context

三轮修复后仍无效。通过阅读 `basic-ftp` 源码发现根因：

- `basic-ftp` 的 `Client.close()` 关闭的是控制连接（control socket）
- 数据传输通过独立的 `dataSocket` 进行
- `close()` 不一定中断活跃的 data socket
- 需要直接 `this.client.ftp.dataSocket?.destroy()` 来中断数据传输

## 修复方案

在 FTPClient/SFTPClient 中新增 `cancel()` 方法，直接销毁底层 data socket。download 的 signal abort 调用 `cancel()`。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/main/services/ftp-client.ts` | TransferClient 新增 `cancel()` 方法；FTPClient.cancel 销毁 dataSocket + close；SFTPClient.cancel 调用 end |
| `src/main/ipc/index.ts` | cancelClient 改为调用 `client.cancel()` |

---

## ftp-client.ts

### TransferClient 接口新增

```ts
cancel(): void
```

### FTPClient.cancel()

```ts
cancel(): void {
  try { this.client.ftp.dataSocket?.destroy(new Error('下载已取消')) } catch {}
  try { this.client.close() } catch {}
}
```

### SFTPClient.cancel()

```ts
cancel(): void {
  try { this.client.end() } catch {}
}
```

### download 方法 — signal abort 调用 cancel()

```ts
if (signal) {
  const onAbort = () => { try { this.cancel() } catch {} }
  if (signal.aborted) { onAbort() }
  else { signal.addEventListener('abort', onAbort, { once: true }) }
}
```

## ipc/index.ts

```ts
await client.download(..., abortController.signal, () => client.cancel())
```

# 第六轮修复（找到根因）

## Context

日志输出：
```
SFTP下载完成: /home/...zip -> /Users/...zip
Error occurred in handler for 'browser:download': ReferenceError: log is not defined
```

**根因：`log` 未在 `ipc/index.ts` 中导入！** 所有 `log.info()` 调用在 catch/finally 块中抛出 `ReferenceError`，导致：
1. catch 块崩溃 → `browser:download-complete` 事件从未发送
2. 渲染进程永远收不到完成事件 → Modal 永远不关闭

## 修复

仅需 1 行改动：在 `ipc/index.ts` 顶部导入 `log`。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/main/ipc/index.ts` | 顶部新增 `import log from 'electron-log'` |

## 验证

1. 下载 → 取消 → 弹框应关闭（或 3 秒兜底关闭）
2. 下载完成 → 弹框正常关闭
3. 控制台无 `ReferenceError` 错误

# 第七轮修复（真正根因）

## Context

日志确认：
- 下载正常完成，`[download] finally block` 执行
- **没有** `[cancel-download] called` 日志 → cancel IPC 从未被调用
- 问题在渲染进程：`handleCancelDownload` 中 `downloadId` 为空

## 根因：React 状态闭包陈旧

`browserDownload` 是 `await` 的，阻塞 ~2 分钟期间组件不会重新渲染。`setDownloadId` 调用后只有当 `await` 返回后组件才 re-render，但 Modal 的 cancel 按钮绑定的是旧 render 的 `handleCancelDownload`，其中 `downloadId` 仍是 `''`。

## 修复：用 useRef 保持最新值

```tsx
const downloadIdRef = useRef('')

// handleDownload 中：
downloadIdRef.current = result.downloadId  // ref 立即更新

// handleCancelDownload 中：
const id = downloadIdRef.current  // 始终获取最新值
```

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/renderer/pages/ServerBrowser.tsx` | 新增 `downloadIdRef`；handleDownload 同步更新 ref；handleCancelDownload 从 ref 读取 |

## 验证

1. 下载 → 点击取消 → 控制台应出现 `[cancel-download] called` 日志
2. 取消后弹框关闭
3. 下载完成 → 弹框正常关闭

# 第八轮修复（真正根因）

## Context

`downloadIdRef` 修复已应用，但 cancel IPC 仍从未被调用。根因：

**`browserDownload` IPC handler 只在下载完成后才 return。** `downloadId` 在 handler 内部创建，但 renderer 的 `await` 要等整个下载结束才能拿到 `downloadId`。下载期间 `downloadIdRef.current` 始终为空。

## 修复

在下载开始前，通过 `webContents.send('browser:download-started', { downloadId })` 立即发送 downloadId。renderer 监听此事件设置 ref。

## 改动文件

| 文件 | 改动 |
|------|------|
| `src/main/ipc/index.ts` | `client.download()` 前发送 `browser:download-started` 事件 |
| `src/main/preload/index.ts` | 新增 `onBrowserDownloadStarted` 监听 |
| `src/renderer/types/ipc.ts` | 新增 `onBrowserDownloadStarted` 签名 |
| `src/renderer/pages/ServerBrowser.tsx` | 监听 `download-started` 事件设置 ref |

## 验证

1. 下载 → 点击取消 → 控制台应出现 `[cancel-download] called`
2. 取消后弹框关闭
